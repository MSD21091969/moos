## Key Decisions Made

### 1. Workspace Structure (Option B)

Each FFS4-8 workspace will have its own `app/` folder alongside `.agent/`:

| Workspace | Purpose                                          | Domain    |
| --------- | ------------------------------------------------ | --------- |
| **FFS4**  | Graph viewer, workspace browser, main agent seat | SIDEPANEL |
| **FFS5**  | PiP WebRTC chat modal for user-user comms        | COMMS     |
| **FFS6**  | IDE workspace container viewer                   | FILESYST  |
| **FFS7**  | ADMIN container viewer                           | ADMIN     |
| **FFS8**  | CLOUD workspace viewer                           | CLOUD     |

**Key insight**: "The workspaces ARE the actual applications." Frontend code is just the visual layer.

### 2. Main Agent Location

- **Main Agent**: Lives in sidepanel (NOT in PiP)
- **Tab Agents**: Managed by ContextManager per-tab
- **Portal app**: ❌ **DELETE** — agents need application context only, not a separate browser tab

### 3. FFS4 as Package

FFS4 `app/` should be an **importable package** bundled with the extension:

```
FFS4_application00_ColliderSidepanelAppnodeBrowser/
├── .agent/           # Context definition
└── app/
    └── package.json  # "@collider/sidepanel-ui"
        └── src/
            ├── GraphViewer.tsx
            ├── WorkspaceBrowser.tsx
            └── MainAgentSeat.tsx
```

## Context Distribution Model

### MainContext (Global)

| Field               | Source                     | Purpose                             |
| ------------------- | -------------------------- | ----------------------------------- |
| `user`              | Firebase Auth → DataServer | Who is logged in                    |
| `applications`      | DataServer `/api/v1/apps`  | All accessible workspaces           |
| `activeApplication` | User selection             | Current workspace context           |
| `container`         | DataServer or FILESYST     | The `.agent` config for current app |

### TabContext (Per-tab)

| Field         | Source                      | Purpose                             |
| ------------- | --------------------------- | ----------------------------------- |
| `tabId`       | Chrome tabs API             | Unique identifier                   |
| `url`         | Chrome tabs API             | Current page URL                    |
| `domain`      | Derived from URL/binding    | FILESYST/CLOUD/ADMIN/SIDEPANEL      |
| `boundAppId`  | User selection or URL match | Which workspace this tab belongs to |
| `domSnapshot` | Content script              | Page structure, selected elements   |
| `agentState`  | Service Worker              | Last action, pending workflow       |

### Context Triggers

| Event                  | Action                                               |
| ---------------------- | ---------------------------------------------------- |
| Tab activated          | Set `activeTabId`, load TabContext, notify sidepanel |
| URL changed            | Update TabContext.url, re-evaluate domain binding    |
| User selects workspace | Update MainContext.activeApplication, load container |
| SSE event              | Update relevant context, broadcast to agents         |

## Connection Architecture

### Protocol Matrix

| Purpose                  | Protocol               | Endpoint                       |
| ------------------------ | ---------------------- | ------------------------------ |
| CRUD operations          | REST                   | DataServer                     |
| Real-time events         | SSE                    | DataServer `/api/v1/sse`       |
| Workflow execution       | WebSocket              | GraphToolServer `/ws/workflow` |
| AI streaming             | WebSocket              | GraphToolServer `/ws/workflow` |
| Tool search              | REST                   | VectorDbServer                 |
| File operations          | Native Messaging       | Python Host                    |
| User-to-user video/audio | WebRTC                 | Peer-to-peer                   |
| WebRTC signaling         | WebSocket              | DataServer `/ws/rtc` (NEW)     |
| Internal UI updates      | chrome.runtime/storage | Extension internal             |

### WebRTC Signaling (Recommendation)

Add to DataServer rather than new server:

```
/ws/rtc (WebSocket)
  - Join room: { type: 'join', roomId, userId }
  - Offer: { type: 'offer', sdp, targetUserId }
  - Answer: { type: 'answer', sdp, targetUserId }
  - ICE: { type: 'ice', candidate, targetUserId }
```

### WebTransport

**NOT RECOMMENDED YET** — Limited Service Worker support. Stick with WebSocket + SSE.

## XYFlow for Workspace Browser

### Why XYFlow (React Flow)

XYFlow is recommended for the workspace/graph browser in the sidepanel:

| Feature                     | Benefit for Collider                                      |
| --------------------------- | --------------------------------------------------------- |
| **Zustand-based state**     | Matches existing extension state management               |
| **Custom nodes/edges**      | Can represent NodeContainers with domain-specific visuals |
| **Multiple handles**        | Supports input/output ports for node relationships        |
| **MiniMap plugin**          | Overview navigation for large workspaces                  |
| **Dagre/ElkJS layout**      | Auto-layout for hierarchical workspace trees              |
| **Fast re-rendering**       | Only changed nodes re-render                              |
| **TypeScript**              | Type-safe integration                                     |
| **Tailwind CSS compatible** | Matches extension styling                                 |

### Implementation Considerations

1. **Node types per domain**:
   - FILESYST: File/folder icon style
   - CLOUD: Cloud/sphere style
   - ADMIN: Cabinet/key style

2. **Integration with ContextManager**:
   - XYFlow nodes represent workspace nodes
   - Selection syncs with `activeApplication`
   - Node data comes from DataServer or FILESYST

3. **Performance for sidepanel**:
   - Sidepanel is narrow — use compact node mode
   - Consider horizontal layout for tree structures
   - MiniMap for navigation in large graphs

4. **Package location**:
   ```
   FFS4/.../app/src/
   ├── WorkspaceBrowser.tsx     # XYFlow canvas wrapper
   ├── nodes/
   │   ├── FilesystNode.tsx     # Custom FILESYST node
   │   ├── CloudNode.tsx        # Custom CLOUD node
   │   └── AdminNode.tsx        # Custom ADMIN node
   └── edges/
       └── ContainerEdge.tsx    # Custom edge with inheritance indicator
   ```

## Architecture Decisions Log

| Decision                        | Rationale                                         |
| ------------------------------- | ------------------------------------------------- |
| Option B (separate app folders) | Cleaner separation, workspace = application       |
| No portal                       | Main agent runs in sidepanel, not browser tab     |
| Package imports (not dynamic)   | Simpler bundling, type-safe, always needed        |
| WebRTC signaling in DataServer  | Keep architecture simple, one user-context server |
| XYFlow for graph browser        | Zustand-based, customizable, performant           |
