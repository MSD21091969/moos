# FFS2 Chrome Extension

> Plasmo-based Chrome Extension in `ColliderMultiAgentsChromeExtension/`.

## Overview

The Chrome Extension is the **user-facing control plane** for the Collider
system. It hosts a three-tab sidepanel: **Compose** (WorkspaceBrowser), **Chat**
(AgentSeat), and **Root** (RootAgentPanel). Agent sessions are composed via the
ColliderAgentRunner and executed by the NanoClawBridge over WebSocket.

## Three-Tab Sidepanel

### Tab 1: WorkspaceBrowser (Compose)

Lets users build a **ContextSet** and start an agent session.

| Control | Purpose |
| ----------------- | ------------------------------------------------------ |
| Role selector | Pick role: superadmin, collider_admin, app_admin, etc. |
| Node multi-select | Select nodes to compose context from |
| Vector query | Optional semantic search for additional tools |
| Compose button | `POST :8004/agent/session` → session_id |

After compose, returns a `SessionResponse` with preview (node_count, skill_count, tool_count)
and `nanoclaw_ws_url`. User switches to the Chat tab.

### Tab 2: AgentSeat (Chat)

Connects to NanoClawBridge via WebSocket and provides the chat interface.

| Component | Purpose |
| ----------------- | ---------------------------------------------------- |
| `nanoclaw-rpc.ts` | WebSocket client to `ws://127.0.0.1:18789?token=...` |
| Chat messages | Streamed via JSON-RPC events |
| Tool results | Displayed inline when agent invokes Collider tools |

**JSON-RPC streaming events:**

| Event | Content |
| ---------------- | ----------------------------------- |
| `text_delta` | Incremental text from LLM |
| `tool_use_start` | Agent invokes a tool (name, params) |
| `tool_result` | Tool execution result |
| `thinking` | Agent reasoning (if enabled) |
| `message_end` | End of agent turn |

### Tab 3: RootAgentPanel (Root)

Auto-composes from `Application.root_node_id` as superadmin:

1. Calls `POST :8004/agent/root/session { app_id }`
2. AgentRunner fetches root_node_id, bootstraps full subtree
3. Writes workspace to `~/.nanoclaw/workspaces/collider-root/`
4. Returns `nanoclaw_ws_url` for direct WebSocket chat
5. 15 Collider tools + NanoClaw built-ins (file, exec, browser)
6. 24h session TTL

## Service Worker Architecture

### Lifecycle

```text
SW Install
   │
   ▼
SW Activate → Init ContextManager
   │
   ▼
Auth Check → Login if needed (username/password → JWT)
   │
   ▼
Load Applications → GET /api/v1/apps
   │
   ▼
Connect SSE → GET /api/v1/sse
   │
   ▼
Ready → Listening for messages + SSE events
```

### Message Routing

The SW routes messages between the sidepanel, content scripts, and external
services:

```text
Sidepanel  ──chrome.runtime.sendMessage──►  Service Worker
                                                │
Content Script ──chrome.runtime.sendMessage──►  │
                                                │
                                    ┌───────────┴───────────┐
                                    │  Message Router       │
                                    │  (type-based dispatch)│
                                    └───────────┬───────────┘
                                                │
                          ┌─────────┬───────────┼───────────┐
                          ▼         ▼           ▼           ▼
                     REST API     SSE        WebRTC     Native
                    (DataServer)  Handler    Handler    Handler
```

### Background Agents

The `background/agents/` directory contains domain-specific agents for direct tool
execution (independent of NanoClaw sessions):

| Agent | Domain | Tools |
| ---------------- | -------- | ---------------------------------------------- |
| `filesyst-agent` | FILESYST | Native Messaging (read/write/list files, sync) |
| `cloud-agent` | CLOUD | REST API (DataServer CRUD) |
| `dom-agent` | — | Content script injection, DOM manipulation |

These agents supplement NanoClaw-based sessions for operations that require
direct browser or filesystem access.

## End-to-End Flows

### Flow 1: Compose + Chat (ContextSet session)

```text
1. User opens sidepanel → WorkspaceBrowser tab
2. User selects role, picks nodes, optionally enters vector query
3. User clicks Compose
4. Sidepanel → POST :8004/agent/session { role, app_id, node_ids, vector_query }
5. AgentRunner:
   a. Authenticates as role via DataServer
   b. Bootstraps each node (GET :8000/api/v1/agent/bootstrap/{id})
   c. Merges contexts (leaf-wins)
   d. Optional: vector search for additional tools
   e. Writes workspace files → ~/.nanoclaw/workspaces/collider/
   f. Returns { session_id, preview, nanoclaw_ws_url }
6. User switches to AgentSeat tab
7. AgentSeat connects ws://127.0.0.1:18789?token=...
8. User sends message → agent.request JSON-RPC
9. NanoClaw reads workspace files, runs agent with composed context
10. Agent streams response (text_delta, tool_use, tool_result, message_end)
```

### Flow 2: Root Agent Session

```text
1. User opens RootAgentPanel tab
2. Panel → POST :8004/agent/root/session { app_id }
3. AgentRunner:
   a. Fetches Application.root_node_id from DataServer
   b. Authenticates as superadmin
   c. Bootstraps full subtree (all descendants)
   d. Writes to ~/.nanoclaw/workspaces/collider-root/
   e. Returns { session_id, nanoclaw_ws_url }
4. Panel connects WebSocket, begins chat
5. Agent has 15 Collider tools + NanoClaw built-ins
6. Session has 24h TTL
```

### Flow 3: Tool Execution (during chat)

```text
1. Agent decides to invoke a tool (e.g. list_nodes)
2. NanoClaw → gRPC :50052 ExecuteTool("list_nodes", { app_id: "..." })
3. GraphToolServer looks up tool in ToolRegistry
4. ToolRunner.execute() → importlib → code_ref Python function
5. Result JSON returned to agent
6. Agent includes result in response → streamed to AgentSeat
```

## Native Messaging

Used by FILESYST domain for local file system operations:

```text
Extension SW ──chrome.runtime.sendNativeMessage──► Native Host (stdio)
                                                          │
                                                    Execute FS ops
                                                          │
Extension SW ◄────────────────── JSON response ───────────┘
```

Supported actions: `read_file`, `write_file`, `list_directory`, `sync_workspace`, `delete_file`.

## Appnode Delivery

When a node is selected, the extension determines the correct frontend appnode:

```text
1. metadata_.frontend_app = "ffs6"
2. Extension opens/navigates to ffs6 frontend at correct route
3. ffs6 receives workspace context + renders appropriate view
```

Core flow: **Agent controls DataServer → Extension delivers appnode → Appnode
renders workspace.**
