# FFS2 Chrome Extension

> Plasmo-based Chrome Extension in `FFS2_ColliderBackends_MultiAgentChromeExtension/`.

## Overview

The Chrome Extension is the **control plane** — it runs agents in the Service Worker, manages application context, and delivers the correct appnode (from FFS3) to the user via the sidepanel.

## Package & Build

| Item            | Value                                          |
| --------------- | ---------------------------------------------- |
| Framework       | Plasmo                                         |
| Chrome manifest | v3 (generated by Plasmo)                       |

## Service Worker Architecture

### Lifecycle

```
SW Install
   │
   ▼
SW Activate → Init ContextManager
   │
   ▼
Auth Check → Login if needed (username/password → JWT)
   │
   ▼
Load Applications → GET /api/v1/apps
   │
   ▼
Connect SSE → GET /api/v1/sse
   │
   ▼
Ready → Listening for messages + SSE events
```

### Message Routing

The SW routes messages between the sidepanel, content scripts, agents, and external services:

```
Sidepanel  ──chrome.runtime.sendMessage──►  Service Worker
                                                │
Content Script ──chrome.runtime.sendMessage──►  │
                                                │
                                    ┌───────────┴───────────┐
                                    │  Message Router       │
                                    │  (type-based dispatch)│
                                    └───────────┬───────────┘
                                                │
                          ┌─────────┬───────────┼───────────┬────────┐
                          ▼         ▼           ▼           ▼        ▼
                     Agent      REST API      SSE        WebRTC   Native
                    (LangGraph) (DataServer)  Handler    Handler  Handler
```

### Message Types

| Type                   | Direction              | Handler              |
| ---------------------- | ---------------------- | -------------------- |
| `LOGIN`                | Sidepanel → SW         | Auth handler         |
| `FETCH_APPS`           | Sidepanel → SW         | API client           |
| `SELECT_APP`           | Sidepanel → SW         | ContextManager       |
| `FETCH_TREE`           | Sidepanel → SW         | API client           |
| `SELECT_NODE`          | Sidepanel → SW         | ContextManager       |
| `AGENT_QUERY`          | Sidepanel → SW → Agent | LangGraph dispatcher |
| `AGENT_RESPONSE`       | Agent → SW → Sidepanel | LangGraph result     |
| `CONTEXT_UPDATE`       | SSE → SW → Sidepanel   | SSE handler          |
| `NODE_MODIFIED`        | SSE → SW → Sidepanel   | SSE handler          |
| `TOOL_EXECUTE`         | Agent → SW → Native    | Native handler       |
| `TOOL_RESULT`          | Native → SW → Agent    | Native handler       |
| `RTC_OFFER/ANSWER/ICE` | SW ↔ WebRTC            | RTC handler          |

## ContextManager

State machine that tracks the active application, selected node, and hydrated context.

### State

```typescript
interface ContextState {
  user: AuthenticatedUser | null;
  applications: Application[];
  selectedAppId: string | null;
  selectedApp: Application | null;
  tree: AppNodeTree[];
  selectedNodePath: string | null;
  selectedNode: AppNode | null;
  hydratedContext: HydratedContext | null;
  connectionStatus: "disconnected" | "connecting" | "connected";
}
```

### State Transitions

```
IDLE → LOGIN → AUTHENTICATED → APPS_LOADED → APP_SELECTED → TREE_LOADED → NODE_SELECTED → CONTEXT_HYDRATED
```

When `NODE_SELECTED`, the ContextManager:

1. Fetches node context via `GET /api/v1/context/:app_id/:node_path`
2. Reads `metadata_.frontend_app` to know which appnode renders this node
3. Layers context: Base → Node container → App config → User ADMIN container
4. Makes tools/instructions/rules available to the active agent

## Agent System

### Browser Agents (LangGraph.js)

| Agent            | Domain   | Tools                                          |
| ---------------- | -------- | ---------------------------------------------- |
| `filesyst-agent` | FILESYST | Native Messaging (read/write/list files, sync) |
| `cloud-agent`    | CLOUD    | REST API (DataServer CRUD)                     |
| `dom-agent`      | —        | Content script injection, DOM manipulation     |

Agents run in the Service Worker. They:

1. Receive user query via `AGENT_QUERY` message
2. Read hydrated context from ContextManager
3. Execute tool calls (REST, Native Messaging, etc.)
4. Stream results back via `AGENT_RESPONSE` message

### Agent Routing

```
User query → Agent Router
                │
                ├── Check active node's domain config
                │
                ├── FILESYST → filesyst-agent (native messaging tools)
                ├── CLOUD   → cloud-agent (REST API tools)
                └── DOM     → dom-agent (content script tools)
```

## Appnode Delivery

The extension is responsible for delivering the correct **appnode** to the user. When a node is selected:

```
1. User selects node in sidepanel (NodeTree)
2. SW receives SELECT_NODE message
3. ContextManager fetches node + hydrates context
4. metadata_.frontend_app = "ffs6" (example)
5. Extension opens/navigates to ffs6 frontend at correct route
6. ffs6 receives workspace context + renders appropriate view
```

This is the core flow: **Agent controls DataServer → Extension delivers appnode → Appnode renders workspace.**

## Native Messaging

Used by FILESYST domain for local file operations:

```
Extension SW ──chrome.runtime.sendNativeMessage──► Native Host (stdio)
                                                          │
                                                          ▼
                                                   Execute FS ops
                                                          │
                                                          ▼
Extension SW ◄──── JSON response via stdout ──────────────┘
```

### Message Format

```json
{
  "action": "read_file",
  "payload": {
    "path": "D:/project/.agent/manifest.yaml"
  }
}
```

Response:

```json
{
  "success": true,
  "data": {
    "content": "..."
  }
}
```

Supported actions: `read_file`, `write_file`, `list_directory`, `sync_workspace`.
