# Critical Research Review: Collider Skills + Runtime Integration

Based on recent (Feb 2026) external research on AI agent architectures, frameworks, and performance benchmarks, here is a critical review of the current `collider-skills-runtime-integration.md` plan.

## 1. Validation of the Current Plan

The external research overwhelmingly validates the core tenets of the Collider integration plan.

### A. The "Harness is the OS" & "Build for Deletion"

**External Finding:** Research on systems like Manus and Claude Code shows that raw LLMs struggle with long-running workflows without a structured environment (intent capture, execution loop, memory). However, the trend is toward **simpler harnesses with smarter models**.
**Plan Alignment:** **STRONGLY ALIGNED.** Choosing the PI framework (created by Mario Zechner) is validated by its description as a "minimalist and highly extensible terminal coding harness." Its core packages (`pi-ai`, `pi-agent-core`) prioritize extensibility over opinionated bloat. The plan's "build for deletion" strategy perfectly matches the industry trajectory.

### B. "Skills are Interface" & Context Mounting

**External Finding:** The **SkillsBench** evaluation framework research proves that "curated skills" significantly enhance agent performance (+16.2% pass rate), while self-generated skills harm it. Crucially, skills are defined as **"context mounting" or "progressive disclosure"**—loading specific knowledge only when needed to prevent context window bloat. Enlarging context windows without management just delays failure.
**Plan Alignment:** **STRONGLY ALIGNED.** The Collider plan's strategy to compute skills dynamically based on the current `NodeContainer` and inject only the top 2-3 relevant skills is exactly what SkillsBench recommends. Combining namespace + version + scope for deterministic skill ranking is the correct approach to context optimization.

## 2. Potential Blind Spots and Recommendations

While the core plan is sound, the research highlights a few areas the current architecture document might be underestimating or omitting.

### A. Sub-Agent Orchestration (The "Manus" pattern)

**Finding:** Advanced agent architectures like Manus and Claude Code rely heavily on a multi-agent system where a high-level orchestrator delegates specialized sub-tasks to sub-agents (e.g., an "Explore" agent vs. a "Plan" agent).
**Critique of Plan:** The Collider plan focuses heavily on the `IAgentSession` and dynamic skill injection for a _single_ agent instance traversing the graph.
**Recommendation:** We need to explicitly define how PI's extension system will handle sub-agent delegation. Does the `ColliderGraphToolServer` expose a `spawn_subagent` tool that the PI runtime natively understands, or does the PI orchestrator handle this internally? The plan should specify if the system is strictly single-agent-multi-skill or multi-agent-multi-skill.

### B. The "CodeAct" Paradigm vs. Static Tools

**Finding:** Manus differentiates itself using the "CodeAct" approach—using executable Python scripts dynamically generated by the agent as the primary action mechanism, rather than relying solely on pre-defined JSON schema tools.
**Critique of Plan:** The plan leans heavily on formalizing contracts (Option A stabilization) and strict JSON schemas for tools via gRPC/MCP.
**Recommendation:** While strict contracts are necessary for stability (Phase 0), we should ensure the PI runtime isn't artificially constrained. The harness should ideally support an "execute_script" or "bash" capability (which PI natively provides via `pi-coding-agent`) to let the model bridge gaps between strict DB tools.

### C. Trajectory and Memory Management

**Finding:** Robust memory systems (like filesystem-as-memory, and preserving "wrong turns" in context for error recovery) are critical for long-running workflows to avoid repeated mistakes.
**Critique of Plan:** The plan focuses on stateless computation of skills from the DB graph.
**Recommendation:** The `PISessionAdapter` must ensure it captures the agent's trajectory (including failed tool calls) and stores it durably, perhaps relying on the `.agent/knowledge/devlog` structure mentioned in the workspaces, rather than just querying the DB for the current 'truth'.

## Conclusion

The architecture plan is excellent and ahead of the curve regarding context management and harness simplicity. The immediate next step (Phase 0: Contract Stabilization) remains the absolute highest priority. The recommendations above primarily affect how we configure the PI runtime _after_ the contracts are stable.
